## 1. Context

### 1.1 Why Layer 2 Meta-Skills

oprim (Layer 1) 提供了 31 个 atomic ops——单个数学操作。但金融分析中有一组**通用工作流**，需要组合多个 atomic ops 才能完成：

- bootstrap Sharpe 分布（bootstrap + sharpe）
- PSR/DSR with skewness adjustment（bootstrap + skew_kurt + sharpe）
- Walk-Forward Optimization（purge_embargo + rolling_window）
- CPCV pipeline with path reconstruction（purge_embargo + bootstrap + summary）
- Distribution shift detection（KS + Wasserstein + JSD ensemble）
- Historical analogy search（DTW + Wasserstein + cosine ensemble）
- Calibration analysis（brier + percentile_rank + bayes_beta）
- ...

如果每个调用方项目自己组合 ops 实现这些工作流，会出现：
- **组合逻辑重复**（多个项目实现相同 walk-forward）
- **集成 bug 重复**（多个项目踩同样的坑）
- **测试覆盖不一致**（某项目测试少，bug 多）
- **学术正确性参差**（CPCV 路径重组实现复杂，易错）

**oskill 的定位**：通用金融分析工作流库，独立于任何业务项目，任何项目都可调用。

### 1.2 这与 oprim / 业务项目的关系

```
                调用方
                  ↓
  ┌───────────────────────────────┐
  │  oskill (Layer 2)       │ ← 本 ADR
  │  组合 Layer 1 完成完整工作流    │
  └─────────────┬─────────────────┘
                ↓ 仅调用
  ┌───────────────────────────────┐
  │  oprim (Layer 1)          │ ← ADR-061
  │  31 atomic ops                 │
  └─────────────┬─────────────────┘
                ↓ 仅调用
  ┌───────────────────────────────┐
  │  Layer 0 标准库                 │
  │  numpy / scipy / pandas / ...  │
  └───────────────────────────────┘
```

**oskill 设计原则**：
- oskill 是 oprim 的**第一批用户**——也是 oprim API 稳定性的最严格 dogfood
- 任何 oprim API 不便都会暴露在 oskill 实现中
- 任何金融分析项目（Helios / Helixa / Tide / Selene / 未来项目）都可同时 pip install oprim + oskill

### 1.3 严格定义：什么是 Layer 2 Meta-Skill

```yaml
layer_2_definition:

  inclusion (必须全部满足):
    ✅ 组合 ≥ 2 个 oprim atomic ops 完成完整工作流
    ✅ 仅依赖 oprim + 标准库 (不调其它 Layer 2)
    ✅ 不调用其它 oskill.* element
    ✅ 在金融分析中是常用通用工作流（不绑特定项目）
    ✅ 单独抽出来比"调用方自己组合 ops"更有价值
    ✅ 抽象层次有"语义价值"——表达一个可命名的工作流

  exclusion (任一即排除):
    ❌ 内部需要调用其它 oskill.* → 退化为业务模块层
    ❌ 业务逻辑专用（e.g. Helios Fusion Score 公式）→ 业务模块
    ❌ 仅 1 项目使用 → 不抽
    ❌ 简单 2 步组合（调用方自己写 5 行代码就行）→ 不抽
    ❌ 必须依赖项目特定 schema → 抽象不到位

  positive_examples:
    skills.bootstrap_sharpe          # 组合 bootstrap_ci + sharpe_ratio
    skills.psr_dsr                   # 组合 3 个 ops, 学术公式严格
    skills.cpcv_pipeline             # 复杂组合 + 路径重组逻辑

  excluded_as_business_module:
    helios_fusion_score_full_pipeline    # Helios 业务专用
    helixa_strategy_decay_state_machine  # Helixa 业务专用
```

## 2. Decision

### 2.1 创建独立 `oskill` package, 包含 13 个 Layer 2 元 skill

```
仓库: github.com/helios-plat/oskill (独立仓库, 与 oprim 平行)
包名: oskill (PyPI 风格命名)
import 名: oskill (Python 模块名)
发布渠道: GitHub Packages (与 helios-plat 同源)
版本: 1.0.0 (semantic versioning, 独立于 oprim)
license: TBD (与 oprim 一致, 推荐 MIT)
依赖: oprim>=1.0.0,<2.0.0 (major version pin)
```

### 2.2 三大架构纪律（与 oprim 一致 + Layer 2 特有）

```yaml
discipline_1_layer_2_no_internal_calls:
  rule: oskill 内部 skill 之间不允许互相 import
  enforcement:
    - PR lint check: 任何 oskill/*.py 文件 import oskill.* → fail
    - CI 阻断 merge

discipline_2_must_use_oprim:
  rule: oskill 内部 skill 必须 import oprim.* 实现核心逻辑
  enforcement:
    - 不允许绕过 Layer 1 直接用 scipy.stats.bootstrap 等被 oprim 已包装的功能
    - PR lint warning: 检测 oskill/*.py 直接 import scipy.stats.bootstrap / bootstrap-like methods
    - code review 强制
  rationale:
    oskill 的价值就在于"严格基于 Layer 1"
    一旦绕过 Layer 1, 出现两套 bootstrap 实现, 一致性丢失

discipline_3_neutral_design:
  rule: API 不假设调用方业务上下文
  enforcement:
    - 文档示例用通用金融场景, 不绑特定项目
    - 输入输出 schema 通用 (DataFrame / dict / np.ndarray)
    - 不引入业务术语 (e.g. Fusion Score / Position Regime Beta)
```

### 2.3 库的独立性原则（继承 oprim, 强化）

```yaml
independence_principles:

  no_business_assumption:
    oskill 不假设任何调用方业务上下文
    API 设计基于"金融分析通用工作流", 不基于"Helios 22 章特定需求"

  no_runtime_dependency:
    不读环境变量 / 配置文件 / 数据库 / 网络
    pure Python functions only

  no_business_logic:
    每个 skill 是组合工作流, 不含业务概念
    业务概念由调用方组合 skills + ops 构造

  no_layer_coupling:
    oskill release 与 oprim 节奏解耦
    oprim 1.x → 2.0 时, oskill 决定何时 migrate
    oskill 不强制调用方升级 oprim

  versioning_independence:
    oskill 1.0 / 1.1 / 2.0 ... 独立演化
    semantic versioning
    pin oprim major version (e.g. >=1.0.0,<2.0.0)
```

#### 3.2 `skills.psr_dsr`

**用途**：PSR + DSR (含 N_eff for DSR)。

**调用 oprim**:
- `oprim.bootstrap_ci`（CI 估计 - 可选）
- `oprim.skew_kurt_robust`（PSR 必需 Fisher-Pearson 校正）
- `oprim.sharpe_ratio`（基础统计量）

**数学定义**:
```
PSR (Probabilistic Sharpe Ratio):
  PSR(SR*) = Φ((SR_obs - SR*) × √(T-1) / √(1 - γ_3 × SR_obs + (γ_4 - 1) / 4 × SR_obs²))
  γ_3 = skewness (bias=False)
  γ_4 = kurtosis (bias=False, excess Fisher)

DSR (Deflated Sharpe Ratio):
  SR_threshold = E[max{SR_i}] for N tested strategies
  ≈ √(2 ln N) - (γ_E + ln(ln N)) / (2√(2 ln N))   (Euler-Mascheroni γ_E ≈ 0.5772)
  DSR = PSR(SR_threshold)
```

**API**:
```python
def psr_dsr(
    returns: np.ndarray,
    *,
    benchmark_sharpe: float = 0.0,
    n_strategies_tested: int | None = None,
    n_eff: float | None = None,
    annualization_factor: float = 252.0,
    bootstrap_ci: bool = False,
    n_bootstrap: int = 1000,
) -> dict
```

**返回**:
```python
{
    "psr": float,
    "psr_ci": (float, float) | None,
    "dsr": float | None,
    "sharpe_observed": float,
    "sharpe_observed_annualized": float,
    "skewness": float,
    "excess_kurtosis": float,
    "n_obs": int,
    "n_eff_used": float | None,
    "warnings": list[str],
}
```

**学术参考**: Bailey & López de Prado 2012, 2014

**跨项目用例**:
- Helios §9.3-9.4 PSR + DSR
- Helixa over-fit 检测
- 任何量化策略验证

#### 3.6 `skills.cpcv_pipeline`

**用途**：Combinatorial Purged CV 完整 pipeline（含路径重组）。

**调用 oprim**:
- `oprim.purge_embargo_split`
- `oprim.bootstrap_ci`
- `oprim.distribution_summary`

**核心逻辑**:
```
n_folds: 总 fold 数 (e.g. 6)
n_test_groups: 每次取多少 fold 作为 test (e.g. 2)
n_combinations = C(n_folds, n_test_groups) (e.g. C(6,2) = 15)
n_paths = n_combinations × (n_test_groups / n_folds)
```

**API**:
```python
def cpcv_pipeline(
    n_total: int,
    *,
    n_folds: int = 6,
    n_test_groups: int = 2,
    label_horizon: int = 0,
    embargo_pct: float = 0.01,
    backtest_fn: Callable | None = None,
    compute_path_statistics: bool = True,
) -> dict
```

**返回（splits_only）**:
```python
{"splits": [...], "n_combinations": int, "n_paths": int}
```

**返回（full_pipeline，给定 backtest_fn）**:
```python
{
    "splits": [...],
    "n_combinations": int,
    "n_paths": int,
    "paths_returns": np.ndarray,
    "paths_sharpe_distribution": dict,
    "paths_sharpe_ci": (float, float),
    "median_sharpe": float,
    "min_sharpe": float,
    "max_sharpe": float,
}
```

#### 3.12 `skills.regime_transition_analysis`

**用途**：Regime transition 完整分析（转移矩阵 + duration + 稳态 + half-life）。

**调用 oprim**:
- `oprim.regime_transition_matrix`
- `oprim.regime_filter_data`
- `oprim.distribution_summary`

**核心计算**:
```
expected_holding_period (per regime) = 1 / (1 - p_stay)
half_life (per regime) = ln(0.5) / ln(p_stay)
```

**API**:

### 组 5：预测质量（Prediction Quality）—— 1 个

#### 3.13 `skills.calibration_analysis`

**用途**：预测概率的完整 calibration 分析（Brier 三分量 + reliability diagram + ECE + MCE）。

**调用 oprim**:
- `oprim.brier_score_decomposed`
- `oprim.percentile_rank`
- `oprim.bayes_beta_update`

**核心计算**:
```
ECE = Σ_k (n_k / N) × |avg_prediction_k - avg_outcome_k|
MCE = max_k |avg_prediction_k - avg_outcome_k|
```

**API**:
```python
def calibration_analysis(
    predictions: np.ndarray,
    outcomes: np.ndarray,
    *,
    n_bins: int = 10,
    binning: Literal["equal_width", "equal_freq"] = "equal_width",
    include_reliability_diagram: bool = True,
    include_bayesian_ci: bool = True,
    prior_alpha: float = 1.0,
    prior_beta: float = 1.0,
) -> dict
```

**返回**:
```python
{
    "brier_score": float,
    "reliability": float,
    "resolution": float,
    "uncertainty": float,
    "skill_score": float,
    "ece": float,
    "mce": float,
    "n_bins": int,
    "binning": str,
    "reliability_diagram": pd.DataFrame | None,
    "n_obs": int,
}
```

**学术参考**: Murphy 1973 + Naeini-Cooper-Hauskrecht 2015 (ECE)

**跨项目用例**:
- Helios §15 Bandit feedback
- Helios §16.6 thesis Brier Score
- Helios §19 prediction quality
- Helixa 信号准确性评估
- 任何概率预测系统

```yaml
cons:
  - 多个项目重复实现相同工作流 (PSR / CPCV / WFO)
  - 路径重组等复杂逻辑易实现错
  - 学术正确性参差

verdict: 拒绝 (与 Layer 2 抽出的初衷矛盾)
```

### 8.2 Alternative B: Layer 2 与 Layer 1 合并成一个 package

```yaml
pros:
  - 单一 package, 安装简单
  - 不用管理依赖

cons:
  - 违反 standalone 原则
  - 调用方可能只需要 Layer 1, 强制带上 Layer 2
  - release 节奏耦合

verdict: 拒绝 (违反三层独立原则)
```

### 8.3 Alternative C: Layer 2 直接用 scipy/sklearn 等, 不强制走 Layer 1

```yaml
pros:
  - 实现更直接
  - 性能可能更好 (少一层)

cons:
  - 失去 Layer 1 的"统一抽象层"价值
  - 同样 bootstrap 在 Layer 1 + Layer 2 各一份, 不一致
  - 违反 oskill 是 oprim dogfood 的定位

verdict: 拒绝 (Layer 2 必须 dogfood Layer 1, 否则三层架构没意义)
```

### 8.4 Alternative D: 只抽部分 (3-5 个高频 skill)

```yaml
pros:
  - 工作量小
  - 快速 ship

cons:
  - 13 个候选都是已识别的真需求
  - 部分抽出意味着调用方仍要重复实现 8-10 个

verdict: 拒绝 (Wiki 已确认"都是必须的, 不分优先级")
```

## 9. Consequences

### 9.1 Positive

```yaml
positive_consequences:

  immediate:
    - 13 个工作流集中实现 + 测试 + 文档
    - 学术正确性集中投入 (CPCV / PSR / Brier 等)
    - 不再分散精力到多项目独立实现

  short_term_after_release:
    - 多项目共享相同 skills, 一致性保证
    - 业务模块开发时间进一步减少
    - 集成 bug 减少 (因为 skills 内部用 oprim 已测过)
    - oprim API 通过 dogfood 反而更稳定

  long_term:
    - 三层架构清晰, 每层维护成本低
    - Layer 3 features 可在 Layer 2 之上快速建立
    - 跨项目共享认知 (Helios 的 PSR 经验自动惠及 Helixa)
    - 评估外部项目时, 可精确判断"对方 PSR 实现 vs 我们的"
```

### 9.2 Negative

```yaml
negative_consequences:

  upfront_cost:
    - 3-4 周 sole architect 投入 (Wiki + 3 CC, 在 oprim 完工后)

  dependency_risk:
    - oskill 依赖 oprim 1.0+
    - oprim major bump 时, oskill 必须同步升级

  testing_overhead:
    - 13 skills × 90% 覆盖 + 集成测试 + 学术对照 ≈ 400+ 测试用例
    - 但比业务模块各自实现的 N × 400 个测试少
```

### 9.3 Risks

```yaml
risks_and_mitigations:

  risk_1_oprim_api_unstable:
    description: oprim 1.0 ship 后短期可能 bug fix
    severity: medium
    mitigation:
      - oskill 启动时 pin oprim>=1.0.5 (允许 patch 后再启动)
      - oprim 进入维护模式后再大规模 ship Layer 2

  risk_2_layer_boundary_drift:
    description: 实施过程中某些 skill 模糊边界 (是 Layer 2 还是 Layer 3)
    severity: low
    mitigation:
      - 严格按"是否调用其它 Layer 2"判断
      - 业务复杂度高的留给 Layer 3

  risk_3_cc_implementation_quality:
    description: CC 自主实现可能在路径重组等复杂逻辑出错
    severity: medium (cpcv_pipeline / historical_analogy_search 特别注意)
    mitigation:
      - 学术对照测试强制 (López de Prado 论文例子)
      - 复杂 skill 由 Wiki 重点 review
      - 集成测试 mock 验证调用关系

  risk_4_business_modules_bypass_skills:
    description: 业务模块仍图省事自实现, 不用 oskill
    severity: medium
    mitigation:
      - lint rule (建议)
      - code review 关注点
      - 文档清晰说明"哪些功能必须用 oskill"

  risk_5_oprim_breaking_changes:
    description: 未来 oprim 2.0 时, oskill 必须 retrofit
    severity: low (1-2 年内不太可能)
    mitigation:
      - oprim API 设计已考虑长期稳定
      - oskill 跟进 oprim major 时也 major bump
```
